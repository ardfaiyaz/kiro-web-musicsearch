# Artist profile images via track artwork lookup

The change gives `ArtistCard` a real profile image by enriching each artist result with artwork from one of their tracks. The iTunes API doesn't expose artist images directly, so `searchArtists` now issues a parallel `/lookup?id={artistId}&entity=song&limit=1` call per artist, plucks `artworkUrl100` from the first track result, and attaches it to the artist object. The component renders the image in a circular crop using Next.js `Image`, falling back to the original SVG icon when no artwork is available.

**Watch for:** Up to 25 additional network requests fire serially-in-parallel on every artist search, which can spike latency and risk iTunes API rate limiting (confirmed). The artwork used is an album/track cover, not the artist's actual portrait — acceptable trade-off given Apple's API surface, but users will see album art where they might expect a face (confirmed).

## High-level view

The enrichment strategy trades search-response latency for visual richness. Each of the up to 25 artists triggers its own lookup call, all fired concurrently via `Promise.all`. If any individual lookup fails or returns no track, the artist gracefully falls back to no artwork — no error propagates. The 5-minute revalidation window keeps repeat queries fast once warm, but first-hit cold searches pay the full fan-out cost.

The component uses `sizes="56px"` and a 200×200 URL rewrite to keep image payload reasonable for a 56px render slot. However, if a returned artwork URL has a hostname outside the `next.config.ts` allowlist (`is{1-5}-ssl.mzstatic.com`), Next.js will reject it at runtime and the SVG fallback won't trigger because `artworkUrl100` is truthy.

1. **Latency fan-out on cold search** — 25 parallel lookup calls per search adds significant wall-clock time on cold caches. Consider batching or capping the number of enrichment calls (e.g., first 10 artists only), or fetching artwork lazily on the client. (confirmed)

Add a concurrency limiter (e.g., `p-limit`) or accept degraded artwork coverage under load. (likely)
3. **Artwork URL hostname coverage** — iTunes artwork URLs occasionally use hostnames outside `is{1-5}-ssl.mzstatic.com` (e.g., `is6-ssl`, `a1.mzstatic.com`). If a track returns one of these, Next.js `Image` will reject it at runtime with a console error and render nothing (the fallback SVG won't show because `artworkUrl100` is truthy). Consider a broader wildcard pattern `**.mzstatic.com` or handling image load errors in the component. (possible — verify against real API responses)

## Enrichment fan-out and latency cost

`searchArtists` now makes up to 25 additional `fetch` calls inside `Promise.all` before returning results. Each call hits `/lookup?id={artistId}&entity=song&limit=1` — a full API round-trip. On a cold cache (first search, or after the 5-minute revalidation window), the total response time is bounded by the slowest individual lookup, not by the original search alone. For a user typing a search and waiting, this can push perceived latency from ~200ms to 1–2 seconds depending on iTunes backend response times.

The `{ next: { revalidate: 300 } }` cache directive helps on repeat queries, but new artist IDs always pay the cold cost. A more defensive approach would cap the number of enrichment calls (say, first 8–10 artists) or move artwork fetching to the client side where a loading skeleton can hide the latency.

## Rate-limit and failure-mode design

The empty `catch {}` block means individual lookup failures are silent — the artist renders without artwork, a fail-open posture. Under moderate concurrent traffic (multiple users searching simultaneously), the unbounded fan-out could trip iTunes API rate limits. Apple doesn't publish exact thresholds, but community reports suggest limits around 20 requests/second/IP. A burst of 3 users searching simultaneously produces 75 lookup requests in a tight window.

There's no concurrency throttle or jittered retry. If rate limiting is encountered, all 25 lookups for a given search would likely fail simultaneously, producing a search result page with no artwork despite all artists having tracks — a confusing UX degradation with no signal to the user or developer (the catch swallows everything silently without even a `console.warn`).

## Image rendering and fallback gap

The component conditionally renders `Image` when `artworkUrl100` is truthy. If the URL is valid but the hostname isn't in `next.config.ts`'s allowlist, Next.js will throw at image optimization time. The allowlist covers `is1` through `is5`, but not all possible mzstatic subdomains. If this edge case hits, the component won't fall back to the SVG — it'll either show a broken image or throw during SSR depending on the Next.js version. Adding an `onError` handler that nullifies the artwork URL (triggering the SVG fallback) would close this gap.

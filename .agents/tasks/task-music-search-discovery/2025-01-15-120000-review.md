# Music Search & Discovery via iTunes API

A complete search-and-browse music application built on Next.js 16 App Router with server-side data fetching from the iTunes Search API. The landing page provides keyword search with URL-persisted query params, results render in a responsive grid, and each track links to a detail page showing metadata, a 30-second audio preview, and "more from this artist" recommendations. The implementation uses Tailwind v4 custom properties for dark mode, semantic HTML throughout, and a React context for single-playback audio coordination.

**Watch for:** Audio player state synchronization relies on 200ms polling rather than events, which creates a subtle UI lag and an unnecessary interval running on every mounted player instance (confirmed). The `play()` callback in `AudioPlayerContext` never handles the promise rejection from `audio.play()`, which will throw an unhandled promise rejection if the browser blocks autoplay or the URL 404s (confirmed). Recommendation fetches that fail are silently swallowed, making it invisible to users or monitoring when artist lookup breaks (confirmed).

## High-level view

The search flow uses Next.js server components with `searchParams` as a Promise (correct for v16) to fetch results on the server and stream them to the client via Suspense. URL-based state means results are shareable and back-button-friendly, and the 60-second revalidation window provides a light cache without stale data risk.

The audio subsystem coordinates single-playback through a context that stores an `HTMLAudioElement` ref and a `currentlyPlayingId` ref. Because refs don't trigger re-renders, each `AudioPlayer` component polls the ref every 200ms to update its local `isPlaying` state. This creates an O(n) interval cost when many players are mounted on the recommendations page, and introduces up to 200ms of visual desync between pressing play and the UI reflecting the change.

The iTunes API integration is a thin wrapper with `try/catch` and typed responses. `searchTracks` and `getTrackById` throw on failure (caught by error boundaries), while `getArtistTracks` returns an empty array on failure. This asymmetry means a broken recommendations endpoint never surfaces to the user as an error, and never gets reported beyond server-side `console.error`.

## Audio playback coordination and state sync

The `AudioPlayerProvider` holds two refs: `audioRef` (the current `HTMLAudioElement`) and `currentlyPlayingId` (the track ID of whatever is playing). The `play` function pauses any existing element, creates a new `Audio(url)`, calls `.play()`, and registers an `ended` listener that nulls both refs.

`audio.play()` returns a Promise, and no `.catch()` handles rejection. On mobile browsers or in contexts without user gesture, this will produce an unhandled rejection. The fix:

```typescript
audio.play().catch(() => {
  currentlyPlayingId.current = null;
  audioRef.current = null;
});
```

There is also a leak vector: when `play()` is called a second time, the old audio element is paused but its `ended` event listener is never removed. If the old element's `ended` fires for any reason after a new track starts, the callback nulls `currentlyPlayingId.current` and `audioRef.current`, breaking the new track's state. The fix is to either remove the listener from the old element or guard the `ended` callback with an identity check against `audioRef.current`.

On the consumer side, `AudioPlayer` uses `setInterval` at 200ms to read `currentlyPlayingId.current` and derive local state. Every mounted `AudioPlayer` (potentially 10+ on the recommendations grid) runs its own timer. The architectural alternative is to promote `currentlyPlayingId` to reactive state (a `useState` in the context) and expose it as a value rather than a ref, which would make consumers re-render on change without polling.

## Error handling asymmetry in API layer

`searchTracks` and `getTrackById` throw on failure, propagating to the nearest `error.tsx` boundary. `getArtistTracks` catches and returns `[]`. A failed recommendation fetch looks identical to an artist with no other tracks. In production this would mask API outages. `console.error` is present so server logs would show the failure, but nothing surfaces client-side.

## Input length validation gap

The `<input>` has no `maxLength` attribute and the server-side function doesn't truncate. A query of thousands of characters will produce a very long URL to the iTunes API. Adding `maxLength={200}` on the input is a one-line fix.

## What is not tested

There is no test framework in the project. Verification is limited to `npm run lint` and `npm run build`. The following behaviors have no automated coverage: search with special characters, empty API responses, audio playback lifecycle, error boundary rendering, navigation between search and detail, recommendation exclusion logic, and the track ID NaN guard in the detail page.

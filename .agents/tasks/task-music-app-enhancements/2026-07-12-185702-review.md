# 16-feature enhancement batch: search, navigation, discovery, offline (v2)

Five of the seven v1 issues have been addressed: the service worker now guards with `response.ok` and has max-entry eviction for the API cache, FeelingLucky surfaces toast notifications on failure, RecentlyViewed compares the raw localStorage string to avoid double-serialization, PageTransition no longer uses `key={pathname}`, and the KeyboardShortcutsOverlay z-index was bumped above MobileBottomNav. The SplashScreen noop subscribe was confirmed intentional (one-time read, no re-subscription needed).

**Watch for:** RecentlyViewed subscribes to a `recently-viewed-updated` custom event that is never dispatched (confirmed), so same-tab updates won't propagate until a full re-render. The service worker's static asset cache still grows unbounded within a version (confirmed) — only the API cache has eviction. PageTransition's `animate-page-enter` fires on initial mount but has no mechanism to re-trigger on client-side navigations since the key was removed (likely).

## High-level view

The RecentlyViewed subscribe function listens for a `recently-viewed-updated` custom event that `addRecentlyViewed` in `lib/recently-viewed.ts` never fires. The homepage section won't reflect new visits until the component remounts (e.g., a full navigation back to `/`). For client-side soft navigations, the stale snapshot persists.

With `key={pathname}` removed, PageTransition is now a memoized static div that never remounts. The CSS animation triggers once at app startup and never again on client-side navigations — the transition feature is effectively dead code.

1. **Missing custom event dispatch for same-tab recently-viewed sync** — `RecentlyViewed` subscribes to `recently-viewed-updated` events, but `addRecentlyViewed` in `lib/recently-viewed.ts` never calls `window.dispatchEvent(new Event('recently-viewed-updated'))`. The fade-in only fires on the initial hard load. Either restore a per-navigation trigger (e.g., using `usePathname()` to toggle a CSS class) or remove the PageTransition wrapper to avoid dead code in the layout.

## RecentlyViewed same-tab sync gap

The `subscribe` function in `RecentlyViewed.tsx` listens for both cross-tab `storage` events and a same-tab custom event:

```typescript
function subscribe(callback: () => void) {
  window.addEventListener("storage", callback);
  window.addEventListener("recently-viewed-updated", callback);
  return () => {
    window.removeEventListener("storage", callback);
    window.removeEventListener("recently-viewed-updated", callback);
  };
}
```

But `addRecentlyViewed` in `lib/recently-viewed.ts` writes to localStorage without dispatching that custom event. The `storage` event only fires in *other* tabs, not the current one. When a user visits `/artist/123`, the `RecentlyViewedTracker` calls `addRecentlyViewed`, but if they navigate back to the homepage within the same tab, `useSyncExternalStore` won't know the store changed until React happens to call `getSnapshot` during an unrelated re-render.

This is partially masked by Next.js App Router sometimes triggering server component re-renders that remount the client boundary on navigation. But for client-side soft navigations (back button, Link click without server round-trip), the stale snapshot persists.

## Static cache eviction

`trimCache` is applied to `API_CACHE_NAME` after every API response:

```javascript
caches
  .open(API_CACHE_NAME)
  .then((cache) => cache.put(request, clone))
  .then(() => trimCache(API_CACHE_NAME, API_CACHE_MAX_ENTRIES));
```

The main `CACHE_NAME` has no equivalent. Static assets in Next.js are fingerprinted (`_next/static/chunks/abc123.js`), so after a deployment, old chunks remain cached indefinitely within the same service worker version. The `activate` handler cleans up *other* cache names on version bump, but within a single `music-app-v1` lifetime, entries accumulate. Low-severity for typical usage, but worth noting for long-lived PWA sessions.

## PageTransition now a no-op on client navigations

With the key removed, the component is a memoized static wrapper:

```tsx
function PageTransitionBase({ children }: PageTransitionProps) {
  return (
    <div className="animate-page-enter">
      {children}
    </div>
  );
}
export const PageTransition = memo(PageTransitionBase);
```

Since it sits in the root layout and never unmounts, `animate-page-enter` fires exactly once on initial page load. Client-side navigations via `next/link` or `router.push` swap the children prop but don't remount the wrapper div, so the CSS animation never replays. The component is dead code that adds a wrapper div without visual effect on SPA navigations.

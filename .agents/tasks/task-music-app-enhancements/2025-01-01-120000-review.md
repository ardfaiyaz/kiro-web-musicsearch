# 16-feature enhancement batch: search, navigation, discovery, offline

This change adds a dedicated `/search` page, mobile bottom navigation, a splash screen, recently viewed tracking, audio preview indicators in TrackCard, a keyboard shortcuts overlay, enhanced genre browsing, a "Feeling Lucky" random play button, OpenGraph metadata for shareable pages, search loading indicators, "Try searching for:" suggestions, page transitions, persistent last search, a service worker for offline caching, and Web Vitals collection. The approach extracts search out of the homepage into its own route, layers discovery and personalization features on top, and finishes with infrastructure (offline, vitals). The change is architecturally well-separated across client and server boundaries.

**Watch for:** PageTransition forces full remount on every navigation via `key={pathname}` (confirmed) which destroys client state in child trees. The service worker caches API responses indefinitely with no TTL or max-entry eviction (confirmed). RecentlyViewed's `getSnapshot` runs `JSON.parse` + `JSON.stringify` comparison on every React render cycle (confirmed), which will degrade with 12 large items. The `subscribeNoop` pattern in SplashScreen means `useSyncExternalStore` never re-subscribes to state changes (likely intentional but brittle).

## High-level view

The homepage sheds its search responsibilities to `/search`, becoming an editorial-only surface with `ContinueListening`, `RecentlyViewed`, and the editorial feed. SearchBar's `performSearch` now navigates to `/search?q=...`, and filter chips follow. This is a clean separation that gives mobile users a focused search experience.

The PageTransition wrapper in `layout.tsx` uses `key={pathname}` to trigger CSS `animate-page-enter` on every route change. This unmounts and remounts the entire child tree, so any client state held by page-level components is destroyed on navigation. Global state (audio player, favorites, theme) lives above the transition boundary and is unaffected.

The service worker implements cache-first for static assets and network-first for navigation and `/api/` routes. It pre-caches only `/` and `/search`. There is no cache size limit, no TTL-based invalidation, and no `response.ok` check, so error responses can be cached and stale data accumulates on long sessions.

Web Vitals collection uses native PerformanceObserver without external dependencies. The LCP and CLS observers persist for the page lifetime (never disconnected), which is the correct behavior for cumulative metrics.

<details>
<summary>Issues (7)</summary>

1. **PageTransition remount destroys page state** — `key={pathname}` unmounts the entire subtree on every navigation. If any page holds ephemeral client state (scroll position, open dropdowns, partially typed forms), it resets. Consider using CSS-only animation without a key-driven remount, or accept the tradeoff explicitly.
2. **Service worker has no cache eviction** — API responses and navigation HTML are cached without max-entries or TTL. On a long session, stale data accumulates. Add a max-age check in the fetch handler or limit cached entries per route type.
3. **RecentlyViewed getSnapshot performance** — `JSON.stringify` comparison runs on every render cycle. With 12 items each containing artwork URLs, this is non-trivial string work. Use a version counter or compare the raw localStorage string against a cached copy instead.
4. **SplashScreen useSyncExternalStore with noop subscribe** — The subscribe function never actually subscribes to anything, so if sessionStorage is cleared while the component is mounted, the store won't re-evaluate. Fragile if this pattern is reused elsewhere.
5. **FeelingLucky has no error feedback** — When the Apple RSS or iTunes fetch fails, the button silently resets to idle. Users will click repeatedly thinking it's broken. Use the existing toast system to surface a brief "No preview available" message.
6. **Service worker caches error responses** — The handler caches any `response.clone()` including 5xx and 404 responses. A failed API call could be served from cache on the next offline fallback. Add `if (response.ok)` before `cache.put`.
7. **KeyboardShortcutsOverlay z-index collision on mobile** — The overlay uses `z-40` and so does MobileBottomNav. The `bottom-20` offset provides visual clearance, but the z-index tie means DOM order determines stacking if they ever overlap.

## PageTransition via key-driven remount

The `PageTransition` component wraps children with `<div key={pathname} className="animate-page-enter">`. When `pathname` changes, React unmounts the entire previous subtree and mounts a fresh one.

```tsx
function PageTransitionBase({ children }: PageTransitionProps) {
  const pathname = usePathname();
  return (
    <div key={pathname} className="animate-page-enter">
      {children}
    </div>
  );
}
```

On the `/search` page this is masked because the query comes from URL params so SearchBar reconstructs with the correct value. But any page holding local ephemeral state would reset. The `prefers-reduced-motion` CSS correctly disables the animation, but the remount still occurs — the tree still reconstructs without the visual effect.

## Service worker cache strategy gaps

The service worker caches responses unconditionally:

```javascript
fetch(request)
  .then((response) => {
    const clone = response.clone();
    caches.open(CACHE_NAME).then((cache) => cache.put(request, clone));
    return response;
  })
```

This pattern appears in all four fetch handler branches (navigation, API, static, default). It caches 5xx errors, 404s, and rate-limited responses. A transient server error gets cached and served on the next offline fallback. The fix is straightforward: guard with `if (response.ok)` before calling `cache.put`.

The cache also grows without bound within a version. For API routes that change with every query (`/api/spotify/artist?name=...`), a dedicated cache with a max-entries policy or periodic cleanup would prevent unbounded storage use.

## RecentlyViewed snapshot cost

`getSnapshot` in `RecentlyViewed.tsx` parses localStorage JSON and double-stringifies to compare:

```typescript
function getSnapshot(): RecentlyViewedItem[] {
  const items = getRecentlyViewed();
  if (JSON.stringify(items) === JSON.stringify(cachedItems)) {
    return cachedItems;
  }
  cachedItems = items;
  return cachedItems;
}
```

`useSyncExternalStore` calls `getSnapshot` synchronously on every render. With 12 items each containing artwork URLs, this involves parsing ~2KB of JSON and serializing it twice for comparison every time any component above `RecentlyViewed` re-renders. A cheaper approach: cache the raw `localStorage.getItem()` string and compare that directly, only parsing when the string changes.

## FeelingLucky silent failure

} catch {

When the Apple RSS endpoint returns an empty feed or the iTunes lookup finds no track with a `previewUrl`, the button shows "Finding a song..." then resets with zero feedback.

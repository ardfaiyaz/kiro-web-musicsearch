# Comprehensive music discovery platform expansion

A basic music search app grew into a full-featured music discovery platform across 5 feature increments: theme system with glassmorphism UI, enhanced search with autocomplete and voice input, artist/album detail pages with Last.fm integration, a discovery feed with trending content and infinite scroll, and a personalization layer with playlists, favorites, and a dashboard. The approach stores all user state in localStorage through a `useSyncExternalStore`-based context pattern, routes through Next.js App Router server components for data fetching, and uses client components for interactive features.

**Watch for:** localStorage quota handling silently drops writes without user notification (confirmed). Last.fm API key is empty string by default making artist bios dead on arrival without env configuration (confirmed). The `RecentlyPlayedTracker` stores fabricated `ItunesTrack` objects with zeroed-out fields that will produce broken UI if rendered on the dashboard (confirmed). Infinite scroll fetches bypass the server-side genre/year/explicit filters applied to initial results, producing inconsistent filtered lists (confirmed).

## High-level view

The theme system hardcodes `dark` on the server-rendered `<html>` element and corrects on mount, producing a flash for light-mode users since there's no blocking script to set the correct class before paint.

Search autocomplete fires live API requests on every keystroke after a 300ms debounce with no cancellation of stale requests, so results can arrive out of order for fast typists.

Last.fm integration is gated behind `NEXT_PUBLIC_LASTFM_API_KEY` which defaults to empty string, causing all Last.fm-dependent features (bios, similar artists, listener stats) to silently return null with no indication that configuration is missing.

The personalization system manages four localStorage collections through a single shared subscription. Any mutation to any collection causes all four snapshots to recompute and all consumers to re-render regardless of which data they actually use.

Infinite scroll loads subsequent pages via client-side fetch against the raw iTunes API. The server applies genre, year, and explicit filters to the initial page only — scrolling past page 1 in a filtered search surfaces unfiltered results.

4. **localStorage quota exceeded is silently swallowed** — Every `try/catch` in `lib/personalization.ts` eats quota errors. Playlists store full track objects, so quota can fill quickly. Surface a toast or warning when writes fail.
5. **Theme flash on light-mode preference** — Server HTML is hardcoded with `dark` class. Users with light-mode preference see a dark flash before `ThemeProvider` corrects on mount. A blocking inline script or cookie-based approach would prevent the flash.
6. **PersonalizationContext shared subscription causes excess re-renders** — All four `useSyncExternalStore` calls share one `notify` that recomputes all snapshots. Split into per-collection subscriptions if performance becomes an issue.
7. **Autocomplete race condition** — Debounced fetches aren't cancelled when a newer keystroke fires. A slower response can overwrite a more recent suggestion set. Use `AbortController` to cancel stale requests.
8. **Artwork download is cross-origin and won't actually download** — `ShareMenu` creates an anchor with `download` attribute pointing to the iTunes CDN. Browsers ignore `download` for cross-origin resources without `Content-Disposition: attachment`, so this opens in a new tab instead. Proxy the image or use a fetch+blob approach.

## Infinite scroll filter inconsistency

The server component in `page.tsx` applies three post-fetch filters before passing tracks to `InfiniteScrollResults`:

```tsx
let tracks = await searchTracks(query, entity);
tracks = filterByGenre(tracks, genre);
tracks = filterByYear(tracks, year);
tracks = filterByExplicit(tracks, explicit);
```

But `InfiniteScroll.tsx` fetches subsequent pages directly from the iTunes API without any of these filters:

```tsx
const response = await fetch(
  `https://itunes.apple.com/search?${params.toString()}`
);
```

A user filtering for "Jazz" from "2010s" will see a curated first page and then random tracks from page 2 onward. The `InfiniteScroll` component receives no filter props and has no awareness of the active search parameters beyond `query` and `entity`.

## RecentlyPlayedTracker data fidelity

`RecentlyPlayedTracker` constructs a synthetic `ItunesTrack` object when recording playback:

```tsx
const trackData: ItunesTrack = {
  trackId: currentlyPlayingId,
  trackName: trackName,
  artistName: artistName || "Unknown Artist",
  artistId: 0,
  collectionName: "",
  artworkUrl100: artworkUrl || "",
  previewUrl: null,
  primaryGenreName: "",
  releaseDate: "",
  trackTimeMillis: 0,
  ...
};
```

The dashboard's `getMostCommonGenre` aggregates `primaryGenreName` from recently played tracks — but that field is always empty here, so genre insights derived from listening history will always be blank. The `artistId: 0` means "recently played" entries would link to `/artist/0` if anyone tried to navigate from them.

## PersonalizationContext subscription granularity

The `notify` callback recomputes all four snapshots unconditionally:

```tsx
const notify = useCallback(() => {
  recentSnapshot.current = getRecentlyPlayed();
  artistsSnapshot.current = getFavoriteArtists();
  albumsSnapshot.current = getFavoriteAlbums();
  playlistsSnapshot.current = getPlaylists();
  listenersRef.current.forEach((listener) => listener());
}, []);
```

Storing full `ItunesTrack` objects in playlists means `getPlaylists()` deserializes potentially large JSON blobs on every unrelated change. If a user has 10 playlists with 50 tracks each, every "recently played" write triggers parsing ~500 track objects.

## Artwork download relies on cross-origin permissiveness

The `ShareMenu`'s "Download artwork" creates an anchor tag with `download` attribute pointing to an iTunes CDN URL. Browsers ignore the `download` attribute for cross-origin resources without `Content-Disposition: attachment`, so this will open the image in a new tab rather than download it. The UX promise of "download" is misleading.

## Theme hydration flash

The layout hardcodes `className="... dark"` on `<html>`. `ThemeProvider` uses `useSyncExternalStore` with `getServerSnapshot` returning `"dark"`, so SSR and initial hydration match. But on mount, if `localStorage.theme === "light"` or `prefers-color-scheme: light`, the `useEffect` removes the `dark` class, producing a visible flash of dark-to-light.

## Test coverage absence

The project has no test framework installed. The filter logic in `page.tsx` (switch-case year ranges, genre matching, explicit toggling), the personalization layer (localStorage round-trips, cross-tab sync, playlist mutations), and the infinite scroll state machine are all ripe for unit tests. Regressions will be caught only at build time (type errors) or manually.

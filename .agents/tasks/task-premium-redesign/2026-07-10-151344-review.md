# Premium editorial redesign with in-app navigation (v2)

The four issues from v1 are addressed: Discover page cards now link to `/track/[id]` and `/album/[id]`, `.transition-premium` is scoped to transform/box-shadow/border-color/color/opacity, the orphaned `Skeleton.tsx` is deleted, and `track/[id]/error.tsx` uses the same premium inline pattern as `app/error.tsx` with the shared `Header` component.

Watch for:
- **Suggestion chips on the home page are non-interactive** (confirmed) — the "Try: Taylor Swift / The Weeknd / Kendrick Lamar" chips render as `<span>` with hover states but no link or click handler. They look clickable but do nothing.
- **Discover hero song has no link** (confirmed) — the hero section featuring the first trending song doesn't link to `/track/[id]`, even though the remaining trending cards in the grid below it do.

**Verdict**: NEEDS_CHANGES

## High-level view

All v1 issues are resolved cleanly. The Discover page cards (trending songs, new releases, featured albums) are wrapped in `<Link>` components with the correct routes. The `transition-premium` utility now lists explicit properties, eliminating the layout-thrash risk on hover-heavy grids. The orphaned Skeleton file is fully removed with no dangling imports. The track error page matches the global error pattern and uses `Header` for navigation consistency.

Two new minor gaps surfaced in the process: the home page suggestion chips and the Discover hero section both signal interactivity without delivering it. These are small compared to the v1 issues but still represent clickable-looking UI that doesn't navigate anywhere.

## Suggestion chips look interactive but aren't

The home page hero section introduces suggestion chips for quick searches:

```tsx
<span className="rounded-full border border-border px-3 py-1 transition-premium hover:border-foreground/30 hover:text-foreground">
  Taylor Swift
</span>
```

These have border and color transitions on hover, signaling clickability, but they're plain `<span>` elements. They should be `<Link href={`/?q=${encodeURIComponent('Taylor Swift')}`}>` so they actually trigger searches. The pattern exists in the same codebase — the Discover page's Popular Artists section does exactly this with `/?q=${encodeURIComponent(song.artistName)}`.

## Discover hero lacks navigation

The Discover page hero renders the first trending song with large artwork, title, and artist name. Every other trending song card in the grid below is wrapped in `<Link href={`/track/${song.id}`}>`, but the hero section that features the same song more prominently doesn't navigate anywhere. This creates an asymmetry: the less prominent cards are interactive but the most prominent one isn't.

Adding a `<Link>` wrapper around the hero content (or at minimum a "Listen now" CTA that links to `/track/${heroSong.id}`) would fix the gap.

# Premium editorial redesign with in-app navigation fix

A full visual overhaul of the music discovery app, replacing the purple/indigo accent palette with a strict black-and-white color system, adding editorial layouts with cinematic hero sections, glassmorphism effects, and refined typography. Simultaneously, ArtistCard, AlbumCard, and SimilarArtists were changed from external link targets (iTunes, Last.fm) to in-app routes, which was the user's primary functional request.

Watch for:
- **Discover page cards are not navigable** (confirmed) — trending songs, new releases, and featured albums render as `<article>` elements without links, so clicking them does nothing despite looking interactive.
- **`transition: all` performance on scroll-heavy pages** (likely) — the custom `.transition-premium` utility animates all properties including layout, which can cause jank on pages with many cards during scroll or hover.
- **Orphaned ErrorMessage component** (confirmed) — the global `error.tsx` removed its import of `ErrorMessage` and inlined new markup, but the component file still exists and is only used by `track/[id]/error.tsx`.

## High-level view

The color palette swap moves CSS variables from indigo/purple to grayscale in both modes, with `--accent` becoming `#171717` (light) / `#ffffff` (dark). Components that used `bg-accent text-white` now use `bg-foreground text-background` for proper contrast. Dynamic color comes only from album artwork blurs.

The navigation fix converts three components from external `<a>` tags to Next.js `<Link>` with internal routes. ArtistCard goes to `/artist/[id]`, AlbumCard to `/album/[id]`, and SimilarArtists to an in-app search query. Users no longer leave the app when clicking artist/album cards.

The editorial layout pattern is consistent across pages: a full-bleed hero section with blurred artwork background, followed by content in a max-width container. Loading skeletons mirror the new layout structure rather than using the previous `Skeleton` components, which remain in the codebase unused.

The Discover page introduces a hero section for the first trending song but does not make the remaining content cards interactive — they have hover states (scale, shadow, translate) but no wrapping link, making them visual dead ends.

## Discover page cards are visual dead ends

The Discover page renders trending songs, new releases, and featured albums as `<article>` elements with hover effects (`hover:-translate-y-1`, `hover:shadow-xl`, `group-hover:scale-105`) that signal interactivity, but none wrap a `<Link>` or attach an `onClick`. The "Popular Artists" section *does* link to a search query, so the inconsistency is within the same page.

This is the most significant functional gap — the page that's supposed to drive discovery doesn't let users navigate to the content it surfaces.

## SimilarArtists routing trade-off

SimilarArtists routes to `/?q=${artist.name}&filter=artist` rather than `/artist/[id]`, since the Last.fm data doesn't carry an iTunes artist ID. This avoids an extra API lookup at the cost of one additional click — the user sees search results that typically include the artist card linking to the detail page.

## The `transition-premium` utility

```css
.transition-premium {
  transition: all 0.3s cubic-bezier(0.4, 0, 0.2, 1);
}
```

`transition: all` on card containers will inadvertently animate padding, width, or any layout-affecting property. On the Discover page grid with 10+ cards visible, rapid mouse movement can trigger multiple reflows. Scoping to `transform, box-shadow, border-color, color, opacity` would produce the same visual effect without the performance risk.

## MiniPlayer — hardcoded white text coupling

The MiniPlayer uses `glass-dark` with hardcoded `text-white` and `text-white/60`, and its progress bar uses `bg-white/80`. These colors are coupled to the dark glass background rather than the theme variables. If the glass variant were ever changed to a lighter style, the text and progress bar would become invisible. The coupling is acceptable for the current fixed design but creates a maintenance trap.

## Header Favorites lost its visual distinctiveness

The Favorites link switched from `text-red-500` / `hover:text-red-500` to the same neutral `bg-foreground/10 text-foreground` pill used by all nav items. Consistent with the black-and-white palette goal, but removes the affordance that helped users spot their favorites link at a glance.

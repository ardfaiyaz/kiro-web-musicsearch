# Heuristic music recommendations via iTunes Search API

The change adds a "You Might Also Like" section to the track detail page that generates music recommendations by querying the iTunes Search API with multiple derived search terms. A recommendation engine in `lib/recommendations.ts` constructs search strategies from genre mappings, mood keywords extracted from track/album names, and thematic terms, then fetches results in parallel, filters out same-artist tracks, deduplicates, and ranks by genre relevance. The results surface through a new API route and a client component that handles loading, error, and empty states.

**Watch for:** Non-deterministic results from `Math.random()` shuffle defeat the 5-minute cache (confirmed). The recommendation engine silently returns empty arrays on all-failures, which means monitoring won't detect upstream iTunes API degradation (confirmed). The client component introduces a client-side fetch waterfall on every track page load since it can't be server-rendered (likely — architectural trade-off, not necessarily a bug).

## High-level view

The recommendation engine uses a multi-strategy approach: a static genre map provides related search terms, mood keywords are extracted by string-matching a fixed word list against track/album names, and genre-prefixed queries add variety. The strategies are intentionally randomized to produce varied results across page loads, but this randomization conflicts with the caching layer that assumes stable responses for the same input.

The API route sets `Cache-Control: public, s-maxage=300, stale-while-revalidate=600` on responses, assuming deterministic output for the same trackId — which the engine doesn't guarantee due to the random shuffle.

Each recommendation request fires 4-5 parallel requests to the iTunes Search API. Under moderate traffic this could exhaust iTunes' undocumented rate limits (roughly 20 req/IP/min). The `{ next: { revalidate: 300 } }` on individual fetch calls provides Next.js-level caching, but the random strategy selection reduces cache hit rates since the same trackId often generates different search terms.

<summary>Issues (5)</summary>

1. **Random shuffle defeats caching** — `generateSearchStrategies` shuffles genre terms with `Math.random()`, so identical requests produce different API queries and different results. The `Cache-Control: s-maxage=300` header promises CDNs they can cache the response for 5 minutes, but the first request sets a random baseline. Either seed the shuffle deterministically (e.g., hash of trackId) or remove the cache header.
2. **Silent total failure with no observability** — If all iTunes API searches fail, `getRecommendations` returns `{ tracks: [], error: false }` because individual `fetchTracksForTerm` calls swallow errors and `Promise.allSettled` never rejects. The component shows the empty state ("No recommendations available") rather than the error state. Add a heuristic: if all strategies returned zero tracks, set `error: true`.
3. **Client-side waterfall on every page load** — The Recommendations component fetches via `useEffect`, meaning the browser must first render the page, then fire a second round-trip to `/api/recommendations`. This adds latency to every track page visit. Consider server-side fetching in the page component (like `getArtistTracks` already does) or at minimum a `loading.tsx` segment.
4. **Genre map coverage gaps** — The `GENRE_SEARCH_MAP` covers 17 genres but iTunes uses many more (e.g., "K-Pop", "Anime", "Christian & Gospel", "Fitness & Workout"). Tracks with unmapped genres fall through to a raw genre-name search, which may produce poor results. The fallback is present but could be improved with a broader map or a dynamic term derivation.
5. **No request deduplication across strategies** — If two strategies produce the same encoded search term (e.g., "pop" from both the genre map and the mood+genre combination), two identical HTTP requests hit the iTunes API. A `Set` check on generated terms before fetching would save network calls and reduce rate-limit risk.

## Search strategy construction and randomization

The engine builds 4-5 search strategies per request: two randomly selected genre terms from a static map, a mood-keyword-plus-genre composite, a "{genre} new music" query, and a thematic term derived from the track name. The randomization via `Array.sort(() => Math.random() - 0.5)` produces variety across page loads so users don't see the same recommendations every time they revisit a track.

The cost is cache incoherence. The API route's `Cache-Control` header tells CDN/edge caches to serve the same response for 5 minutes. But since the underlying queries change on every invocation, the cached response is just whichever random permutation happened first. Subsequent origin hits will produce different results, creating inconsistency between cached and fresh responses. A deterministic shuffle (seeded by trackId, or by trackId + date-bucket for daily rotation) would preserve variety over time while making the cache meaningful.

The `sort(() => Math.random() - 0.5)` pattern also produces biased shuffles — it doesn't give uniform distribution across permutations. For 5 elements this barely matters for recommendation quality, but it compounds the cache-defeat problem since even the "random" selection isn't reliably random.

## Error propagation asymmetry

The system has two error paths that behave differently:

1. `getTrackById` throws on failure (from `lib/itunes.ts`), which the API route catches and returns as a 500. The client shows the error state.
2. `fetchTracksForTerm` swallows errors and returns `[]`. If all strategies fail, `allTracks` is empty, the function returns `{ tracks: [], error: false }`, and the client shows "No recommendations available" — indistinguishable from a track that genuinely has no similar music.

Partial degradation (some strategies fail, some succeed) is handled gracefully, but total degradation is invisible. The empty-state message implies "we looked and found nothing" when the truth might be "we couldn't look at all." Fix: check if every `Promise.allSettled` result is either rejected or returned an empty array, and set `error: true` in that case.

## Client-side fetch waterfall

The Recommendations component fetches on mount via `useEffect`, meaning the recommendation data doesn't participate in the server-rendered HTML. The page's initial HTML includes the "More from Artist" section (fetched server-side) but shows a loading spinner for AI recommendations until the client-side fetch completes. This creates a visible content shift and adds one full round-trip of latency (browser → own API route → iTunes API → back).

This is the primary UX cost of the architecture. The existing `getArtistTracks` call demonstrates that server-side fetching is viable in this page component. Moving recommendations server-side (or using React Server Components with Suspense boundaries) would eliminate the waterfall.

## Ranking single-signal and ordering bias

Ranking uses a single binary signal: does the result's `primaryGenreName` match the source track's genre? Within each group (match / no-match), insertion order from the parallel fetches is preserved. Tracks from whichever strategy resolves first dominate the top positions among same-genre results, introducing non-deterministic ordering that varies by network timing.

The `artistId` filter may let through same-artist tracks when an artist has multiple IDs in the iTunes database (common for artists who switch labels or have regional variants). A secondary check on `artistName` normalization (case-insensitive, trim whitespace) would close this gap.

## iTunes API rate limiting exposure

Each recommendation request fires 4-5 parallel requests to the iTunes Search API. Under moderate traffic (say, 5 concurrent users browsing tracks), the server could send 20-25 requests per minute to iTunes from a single IP. The Next.js fetch cache (`revalidate: 300`) helps for repeated identical terms, but the random strategy selection means cache hits are unreliable. No backoff or retry logic exists — a rate-limited response (HTTP 403 or 429) is silently swallowed, contributing to the silent-failure problem described above.

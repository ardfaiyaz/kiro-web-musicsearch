# Heuristic music recommendations via iTunes Search API (v2)

The change adds a "You Might Also Like" section to the track detail page, powered by a multi-strategy recommendation engine that queries the iTunes Search API with genre, mood, and thematic search terms derived from the source track's metadata. All five issues from the v1 review have been addressed: the random shuffle is now a seeded PRNG keyed on trackId, total failure is detected via an `anyStrategySucceeded` flag, the client-side fetch waterfall is eliminated by calling `getRecommendations` server-side in the page component, the genre map is expanded from 17 to 32 entries, and search terms are deduplicated via a `Set` before fetching.

**Watch for:** The `anyStrategySucceeded` heuristic has a false-negative gap — fulfilled promises that returned empty arrays (from non-200 responses) count as "not succeeded" only by coincidence of the `length > 0` check, but a 200 response with zero results also registers as "not succeeded," making genuinely-no-results indistinguishable from rate limiting (likely). The API route is now orphaned infrastructure — it duplicates logic already called server-side and adds an unused public endpoint (confirmed). Same-artist filtering relies solely on `artistId`, which may miss multi-ID artists (possible).

**Verdict**: APPROVED

## High-level view

The recommendation engine's deterministic strategy selection (seeded Fisher-Yates shuffle) now produces stable results for a given trackId, meaning the Next.js fetch-level `revalidate: 300` on individual iTunes calls and the API route's `Cache-Control` header both function as intended.

Server-side integration in the page component calls `getRecommendations(track)` alongside the existing `getArtistTracks` call, so recommendations are included in the initial server-rendered HTML with no client-side waterfall. However, the two independent server-side calls are awaited sequentially rather than parallelized.

The API route at `/api/recommendations/route.ts` now exists as dead code — the page never calls it. It's a public, unauthenticated endpoint that proxies iTunes lookups with no rate limiting of its own.

The `anyStrategySucceeded` flag detects total failure better than v1's silent empty-array return, though it conflates "all searches genuinely returned zero tracks" with "all searches failed silently."

## Orphaned API route

The API route at `app/api/recommendations/route.ts` performs a full `getTrackById` lookup plus `getRecommendations` for any numeric trackId passed as a query parameter — meaning anyone can enumerate track recommendations without authentication. It's no longer referenced by any page or component after the move to server-side rendering.

It consumes server resources and counts against iTunes API rate limits if called externally. If the plan is to support client-side fetching in the future, documenting that intent in a comment or gating behind an env-var check would clarify the code's purpose.

## Error detection semantics

The `anyStrategySucceeded` logic:

```typescript
for (const result of results) {
  if (result.status === "fulfilled") {
    if (result.value.length > 0) {
      anyStrategySucceeded = true;
    }
    allTracks.push(...result.value);
  }
}

if (!anyStrategySucceeded) {
  return { tracks: [], error: true };
}
```

A `fulfilled` promise with an empty array means either: (a) the iTunes API returned a valid 200 response with zero matching tracks, or (b) `fetchTracksForTerm` caught an HTTP error and returned `[]`. Both paths look identical here. This means a genuinely-zero-results scenario (valid API response, just nothing matched) reports as an error — the user sees "Unable to load" when a silent empty state would be more appropriate.

A simple improvement: have `fetchTracksForTerm` return a richer type (e.g., `{ tracks: ItunesTrack[], succeeded: boolean }`) so the caller can distinguish "API worked, nothing matched" from "API failed."

## Sequential server-side fetches

The page component awaits two independent calls sequentially:

```typescript
const recommendationsResult = await getArtistTracks(track.artistId, track.trackId);
const aiRecommendations = await getRecommendations(track);
```

`getRecommendations` fires 4-5 parallel iTunes requests internally, so total wall-clock time for the page is: `getArtistTracks` latency + `getRecommendations` latency. Parallelizing with `Promise.all([getArtistTracks(...), getRecommendations(track)])` would reduce TTFB by the duration of whichever call is shorter.

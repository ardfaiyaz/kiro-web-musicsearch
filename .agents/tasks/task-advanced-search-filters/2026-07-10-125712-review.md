# Advanced search filters for iTunes music app

This change adds entity filtering (Song/Artist/Album) and sort controls (Relevance/Release Date/Artist Name) to the music search interface. Filter and sort state lives in URL search params, the SearchFilters component updates them via client-side navigation, and the server component reads them to shape the API call and sort results before rendering. URL params serve as the single source of truth, with server-side sorting and API-level filtering delegated to the iTunes `entity` parameter.

**Watch for:** Artist and Album filters return non-track entities that get cast to `ItunesTrack` without field validation, which will produce runtime property access on undefined fields in TrackCard (confirmed). Sort by `releaseDate` and `artistName` will produce wrong output on these non-track results for the same reason (confirmed).

## High-level view

The entity filter maps user selections to the iTunes API's `entity` parameter (musicTrack, musicArtist, album). When the user selects Artist or Album, the API returns objects whose shape does not match `ItunesTrack` — they lack `trackId`, `trackName`, `trackTimeMillis`, and `artworkUrl100`. The code explicitly bypasses the `wrapperType === 'track'` filter for these cases and casts the results to `ItunesTrack[]`, so TrackCard will render with undefined fields. This is the central defect.

Sorting is applied server-side after the API call. The sort comparators access `releaseDate` and `artistName` directly — both may exist on artist and album result types, so sorting itself won't crash, but the sorted results still render through a component that expects track-shaped data.

The SearchFilters component manages state entirely through URL params and uses `router.push` for navigation, triggering a full server component re-render for each filter change.

<summary>Issues (3)</summary>

1. **Artist/Album entity results cast unsafely to ItunesTrack** — When entity is "artist" or "album", API results lack trackId, trackName, artworkUrl100, trackTimeMillis and other fields TrackCard depends on. Either filter these entities out and only pass tracks, or create separate card components for artist/album result types.
2. **TrackCard crashes on missing fields** — `track.trackId` is used as both the `key` prop and a Link href segment; it will be `undefined` for artist/album results, causing React key collisions and broken navigation. Guard these accesses or prevent non-track entities from reaching TrackCard.
3. **No input validation on filter/sort URL params** — Arbitrary strings in `?filter=` or `?sort=` pass through to the API call or sort logic without validation. A value like `filter=musicVideo` would hit the else branch (no entity param added) and behave like "all", which is safe but inconsistent. Consider validating against the known option sets.

## Entity filter bypasses the wrapperType guard

The pre-existing `searchTracks` function filtered results with `.filter(item => item.wrapperType === 'track')`, ensuring only track-shaped objects reached the UI. The new code adds this branch:

```typescript
if (entity === "artist" || entity === "album") {
  return data.results as ItunesTrack[];
}
```

This early return skips the filter entirely and asserts a type that doesn't hold. The iTunes API returns objects with `wrapperType: "artist"` or `wrapperType: "collection"` for these entity types. These objects have different field sets — an artist result has `artistName` and `artistId` but no `trackName`, `trackId`, `artworkUrl100`, or `trackTimeMillis`. An album result has `collectionName` and `collectionId` but similarly lacks track-specific fields.

The task's own findings note acknowledges this: "artist-only and album-only filters may return empty results for some queries." But the code doesn't return empty results — it returns the non-track objects and casts them, which is worse than returning nothing. TrackCard will attempt to render `track.trackName` (undefined), use `track.trackId` in a Link href (undefined → `/track/undefined`), and call `track.artworkUrl100?.replace(...)` (undefined, handled by the optional chain but still showing the placeholder).

The `key={track.trackId}` in TrackGrid will be `undefined` for every artist result, causing React key collisions and unpredictable rendering behavior.

## Sort comparators assume track-shaped data

The `sortTracks` function accesses `a.releaseDate` and `a.artistName`. Artist API results do have `artistName` but don't have a meaningful `releaseDate` (the field may not exist or may be null). Album results have `releaseDate` and `artistName`. Sorting a mix of entity types produces incoherent ordering since the fields mean different things across types, and `new Date(undefined).getTime()` returns `NaN`, making the comparator non-deterministic.

## No param validation at the page boundary

`activeFilter` is typed as `string` and passed directly to `searchTracks`. Any value that isn't "song", "artist", or "album" falls through all the if/else-if branches without adding an entity param — accidentally equivalent to "all". Similarly, an unrecognized sort value returns tracks unsorted, equivalent to "relevance". This is safe today but fragile: adding new sort/filter values without updating validation will silently degrade to defaults rather than surfacing an error.

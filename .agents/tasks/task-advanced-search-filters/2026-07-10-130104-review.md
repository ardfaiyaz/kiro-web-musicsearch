# Advanced search filters for iTunes music app (v2)

All three issues from the first review have been addressed. `searchTracks` now always requests `entity=musicTrack` and uses the `attribute` parameter (artistTerm/albumTerm/songTerm) to narrow which field the search term matches against. The `wrapperType === 'track'` filter remains in place for all code paths. URL param validation uses `VALID_FILTERS`/`VALID_SORTS` const arrays with type guards, rejecting unrecognized values at the page boundary.

**Watch for:** The `releaseDate` sort comparator doesn't guard against missing or malformed date strings from the API ‚Äî `NaN` from `new Date(undefined).getTime()` makes `Array.sort` non-deterministic (likely).

**Verdict**: APPROVED

## High-level view

The API layer uses `entity=musicTrack` unconditionally and scopes searches via the `attribute` query parameter. Selecting "Artist" tells iTunes "treat the search term as an artist name" rather than requesting artist-type entities, so all responses are track-shaped and the v1 entity mismatch is eliminated.

The `releaseDate` sort comparator calls `new Date(b.releaseDate).getTime()` without a fallback. Track entities reliably have this field, but an absent or malformed value would produce `NaN`, yielding implementation-defined sort order.

The const arrays defining valid filter/sort values exist independently in `page.tsx` (for validation) and `SearchFilters.tsx` (for rendering). Adding a new value to one without the other would cause the UI to offer an option the server silently ignores.

## Release date sort edge case

```typescript
return [...tracks].sort(
  (a, b) =>
    new Date(b.releaseDate).getTime() - new Date(a.releaseDate).getTime()
);
```

If a track has an absent or malformed `releaseDate`, `getTime()` returns `NaN`, and `NaN - number` is `NaN`, which makes `Array.sort` produce implementation-defined ordering. A guard like `new Date(b.releaseDate || 0).getTime()` would clamp these to epoch and push them to the end of a descending sort. Low priority but trivial to add.

## Duplicated option constants

`page.tsx` defines:
```typescript
const VALID_FILTERS = ["all", "song", "artist", "album"] as const;
const VALID_SORTS = ["relevance", "releaseDate", "artistName"] as const;
```

`SearchFilters.tsx` defines:
```typescript
const FILTER_OPTIONS = [
  { value: "all", label: "All" },
  { value: "song", label: "Song" },
  ...
] as const;
```

If a new filter value is added to `SearchFilters.tsx` but not `page.tsx`, the UI will offer an option that the server silently ignores (falling back to "all"). Extracting a shared `lib/constants.ts` with the canonical value lists would eliminate this drift risk.

# Custom music preview player with full playback controls (v2)

The AudioPlayer component was rewritten from a minimal play/pause button into a full-featured custom player: seekable progress bar, current and remaining time display, volume slider with mute toggle, and three-state volume icon. The AudioPlayerContext was extended to expose `currentTime`, `duration`, `volume`, `progress`, `setVolume()`, and `seekTo()`. Route-change cleanup uses `usePathname` to stop audio on navigation. The three issues from the first review (lossy mute toggle, stale volume closure, orphaned event listeners) are all addressed.

**Watch for:** Seeking while paused silently fails (confirmed). The context provider value object is recreated on every state change, causing re-renders of all consumers on every `timeupdate` tick (confirmed). No test coverage for any of the new behavior (confirmed).

**Verdict**: APPROVED

## High-level view

The prior review's three blocking issues are resolved. `previousVolumeRef` stores the pre-mute volume and restores it on unmute. `volumeRef` is read inside `play` so the callback doesn't close over stale volume state. `cleanupRef` holds the listener-removal function and is called before discarding old audio elements or on stop.

The context provider rebuilds its value object on every `currentTime` state update (~4 times/sec during playback), which re-renders all consumers including inactive player instances. For a single-player page this is harmless; for a list view with many mounted players it would cause unnecessary work.

`pause()` destroys the audio element entirely rather than pausing in place. This means seeking while stopped is a no-op (there's no element to seek on) and resuming replays from the beginning. For 30-second previews this is a reasonable tradeoff, but it's a UX asymmetry worth understanding before adding resume-from-position features.

## Pause semantics: stop-and-destroy vs. pause-in-place

Calling `pause()` invokes `stopAudio()`, which destroys the audio element, nulls `audioRef`, removes listeners, and resets all playback state to zero. Resuming after pause replays from the beginning (a new `play()` call creates a fresh `Audio` element from the URL). For 30-second previews this is arguably fine, but it means the progress bar and time display snap to zero immediately on pause rather than holding position. If resume-from-position is added later, this would need a separate `pause` path that calls `audio.pause()` without destroying the element.

## Re-render cascade during playback

```
timeupdate fires (~4/sec)
  -> setCurrentTime(audio.currentTime)
  -> setProgress((currentTime / duration) * 100)
  -> AudioPlayerProvider re-renders
  -> new value={{ play, pause, ..., currentTime, progress, ... }}
  -> all useAudioPlayer() consumers re-render
```

Every mounted `AudioPlayer` instance — including those for tracks not currently playing — re-renders on every tick. The `isPlaying` check prevents them from doing meaningful DOM work (they show static "0:00" / "30-second preview"), but React still runs the component function and diffing. For a page with 20 search results each rendering an AudioPlayer, this is ~80 unnecessary component invocations per second.

A split-context pattern (one context for stable callbacks like `play`/`pause`/`setVolume`, another for volatile state like `currentTime`/`progress`) would let inactive players subscribe only to the stable context.

## Volume edge case on manual zero

If the user drags volume to 0 manually (via the slider, not the mute button), `previousVolumeRef` retains whatever value it had at mount (1). Pressing the "unmute" button then restores to 1 rather than the last non-zero slider position. This is minor — the slider gives continuous feedback so users rarely land exactly at 0 — but it's a behavioral inconsistency between the two paths to silence.

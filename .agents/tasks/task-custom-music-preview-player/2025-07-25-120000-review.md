# Custom music preview player with full playback controls

The AudioPlayer component was rewritten from a minimal play/pause button into a full-featured custom player: seekable progress bar, current and remaining time display, volume slider with mute toggle, and three-state volume icon. The AudioPlayerContext was extended to expose `currentTime`, `duration`, `volume`, `progress`, `setVolume()`, and `seekTo()`, with `timeupdate`/`loadedmetadata` event listeners driving state updates. Route-change cleanup uses `usePathname` to stop audio when the user navigates away. The component API (`previewUrl`, `trackId`, `trackName`) stays backward compatible.

**Watch for:** Mute toggle loses the previous volume level (confirmed). The `play` callback closes over the `volume` state value, causing callback identity churn on every volume change and a potential stale-volume edge case (likely). Event listeners on discarded audio elements are never cleaned up (confirmed).

## High-level view

The context now manages the full `HTMLAudioElement` lifecycle and exposes derived playback state. The route-change cleanup uses a ref-to-function pattern to avoid effect re-subscriptions when `stopAudio` changes identity â€” a non-obvious but sound technique.

The mute/unmute implementation is lossy: muting sets volume to 0, unmuting hard-codes it to 1 regardless of the previous value. There is no "remember previous volume" mechanism.

When switching tracks, `attachListeners` returns a cleanup function that is never called. The orphaned element's listeners remain allocated until GC, referencing stale closures in the meantime.

## Lossy mute toggle

The `handleMuteToggle` function in AudioPlayer.tsx toggles between `setVolume(0)` and `setVolume(1)`. Once muted, the original volume level is gone:

```tsx
function handleMuteToggle() {
  if (volume > 0) {
    setVolume(0); // but where is the old volume stored?
  } else {
    setVolume(1); // should be setVolume(previousVolume)
  }
}
```

A `previousVolumeRef` in the context (or locally in the component) would fix it. Confidence: **confirmed** â€” the code at lines 89-93 of AudioPlayer.tsx explicitly sets 0/1 with no storage.

## `play` callback re-creation and stale volume

The `play` callback in AudioPlayerContext.tsx has `volume` in its dependency array (line 143). Every volume slider interaction creates a new `play` function. If a consumer holds a stale reference to `play` (e.g., from a closure captured before a volume change), the new Audio element will be initialized with the wrong volume.

The fix is to read volume from a ref inside `play`:

```tsx
const volumeRef = useRef(volume);
useEffect(() => { volumeRef.current = volume; }, [volume]);

// Then inside play:
audio.volume = volumeRef.current;
```

This also removes `volume` from the dep array, making `play` stable across volume changes. Confidence: **likely** â€” the callback identity churn is confirmed in the code; whether it manifests as a wrong-volume bug depends on call-site timing.

## Orphaned event listeners on track switch

In the `play` function, when a new track starts, the old `audioRef.current` is paused and nulled:

```tsx
if (audioRef.current) {
  audioRef.current.pause();
  audioRef.current = null;
}
```

But `attachListeners(audio)` returns a cleanup function that is never stored or called. The old audio element still has `timeupdate`, `loadedmetadata`, and `ended` listeners attached. The `handleEnded` guard (`if (audioRef.current === audio)`) prevents state corruption, but the listeners remain allocated until the orphaned `HTMLAudioElement` is garbage collected.

Store the cleanup in a ref and call it before creating a new element:

```tsx
const cleanupRef = useRef<(() => void) | null>(null);

// in play():
cleanupRef.current?.();
cleanupRef.current = attachListeners(audio);
```

Confidence: **confirmed** â€” the return value of `attachListeners` is discarded on line 135 of AudioPlayerContext.tsx.

## Shared playback state across multiple player instances

The context stores a single `currentTime`, `duration`, `progress`, and `volume`. If multiple `AudioPlayer` components render on the same page, every instance reads from the same state. Currently the `isPlaying` guard (`currentlyPlayingId === trackId`) in AudioPlayer.tsx prevents non-active instances from displaying stale values. If a future refactor removes that guard, all player instances would show identical time readouts.

Confidence: **possible** â€” depends on future usage patterns; the current guard prevents visible issues.

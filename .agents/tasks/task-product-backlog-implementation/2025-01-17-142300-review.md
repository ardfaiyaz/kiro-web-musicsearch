# Enhanced music player, discovery browsing, settings page, and UI polish

This change transforms a basic music search app into a full-featured player application across five coordinated features: an upgraded AudioPlayerContext with shuffle/repeat/sleep timer/history/Media Session API, a discovery page with browse-by-mood/decade/country/activity/theme sections, a dedicated settings page replacing the slide-out panel, and a comprehensive UI/UX pass (skeleton loading, context menus, bottom sheets, 404 page, accessibility improvements, memoized cards). The approach is pragmatic: heavy use of localStorage for state persistence, `useSyncExternalStore` for cross-tab sync on theme, a signal-counter pattern for the ended-event to satisfy React 19 compiler rules, and Suspense boundaries around async discovery components.

**Watch for:** Event listener leak in `startPlayingTrack` where `attachCoreListeners` cleanup is never stored or invoked on the hot path (confirmed). Media Session API registers `previoustrack` and `nexttrack` as `null` instead of wiring them to the actual handlers (confirmed). The `play` callback captures stale `currentTrack` in its dependency array, creating a growing closure chain that only clears when the component unmounts (likely). Route-change effect kills playback unconditionally, even for sub-navigations within player-visible routes (confirmed).

## High-level view

The AudioPlayerContext was substantially rewritten from a simple play/pause/queue model to a multi-mode player with shuffle, repeat, previous-track history, sleep timer, mute, playback rate, and localStorage-backed history. The core architectural choice is a counter-increment pattern (`endedCounter`) that decouples the audio `ended` event from the state-heavy logic that decides what plays next, satisfying React 19's strict rules about effects. This works but leaves the ended-effect with suppressed lint rules and stale closure risk on queue and repeat state.

Search was decomposed: the monolithic SearchBar dropdown was split into `SearchSuggestions`, `InstantResults`, and `AdvancedFilters` subcomponents. Discovery got five new server components for browse-by-category cards (pure link grids, no client JS) plus two async components wrapped in Suspense for data-fetching (HiddenGems, PopularThisWeek).

Theme handling moved to a module-level external store (`useSyncExternalStore`) supporting "system" mode, listening to `prefers-color-scheme` changes, and applying a `theme-transitioning` class that forces transitions on all DOM elements during the switch. The inline script in `<head>` was updated for the "system" value so there's no flash. The wildcard `*` transition rule will cause frame drops on element-heavy pages.

The settings panel was demoted to a quick-access overlay with a link to `/settings`, a full page with five independent section components (Appearance, Playback, Privacy, Storage, About), each managing its own localStorage slice.

Accessibility got meaningful structural upgrades: skip links (to content, player, navigation), `aria-current` on nav links, `aria-live` on the mini player for track-change announcements, 44px minimum touch targets, focus trapping in bottom sheet/context menu.

<summary>Issues (8)</summary>

1. **Listener leak in `startPlayingTrack`** — `attachCoreListeners` returns a cleanup function that is never stored on success. Each track transition leaks three event listeners on the discarded Audio element. Store the cleanup ref and invoke it before creating new audio.
2. **Media Session skips previous/next** — `navigator.mediaSession.setActionHandler("previoustrack", null)` and `"nexttrack", null` disable hardware skip controls. Wire them to `previousTrack` and `playNext`.
3. **Route-change kills playback unconditionally** — The pathname effect pauses and clears the player on every route transition, contradicting the persistent mini-player and "continue listening" features. Remove or gate it to specific routes.
4. **Stale `currentTrack` in `play` closure** — `play` depends on `currentTrack` to push to history, but captures the value at callback creation time. Rapid sequential calls may push stale references. Use a ref for the history-push source.
5. **`endedCounter` effect has no deps beyond counter** — The effect reads `queue`, `shuffleMode`, `repeatMode`, `currentTrack`, `startPlayingTrack`, `clearPlayerState` but lists only `endedCounter`. Toggling repeat right before a track ends won't take effect. Use refs or add proper dependencies.
6. **Sleep timer side-effects in state updater** — `setSleepTimerRemaining`'s updater pauses audio and calls `setIsPlaying` (impure). React concurrent mode may call updaters multiple times. Move side effects to a separate effect watching `sleepTimerRemaining === 0`.
7. **ContinueListening subscriber is a no-op** — `useSyncExternalStore` with a no-op subscribe never re-renders when history changes in the same tab. The card shows stale data until remount. Implement a real subscription mechanism (custom event or listener array).
8. **Theme transition wildcard performance** — `theme-transitioning` applies `transition: ... !important` to `*`, `*::before`, `*::after`, forcing the browser to composite transitions on every DOM element for 300ms. On element-heavy pages (discover), this drops frames. Scope the transition to specific properties on the root or use `will-change` selectively.

## Listener leak in `startPlayingTrack`

`startPlayingTrack` calls `attachCoreListeners(audio)` and stores the cleanup in a local `const cleanup`, but only uses it in the play-failure catch branch. On success, the cleanup reference is lost. When the next track starts and `audioRef.current.pause()` is called on the old element, the three event listeners (`timeupdate`, `loadedmetadata`, `ended`) remain attached. For Audio elements that aren't garbage collected immediately (browser media session references), those listeners continue to fire `setCurrentTime`, `setProgress`, and `setEndedCounter` against a stale element.

The `play` function has the same pattern: `attachCoreListeners(audio)` is called but the return value is discarded. The original code used `cleanupRef` to track this; the refactored code dropped it.

```typescript
// startPlayingTrack - cleanup is local and lost on success
const cleanup = attachCoreListeners(audio);
audio.play().catch(() => {
  if (audioRef.current === audio) {
    cleanup(); // only called on failure
    ...
  }
});
// success path: cleanup is never called before next track
```

Fix: restore a `cleanupRef` that stores the current cleanup function and invoke it before attaching new listeners.

## Media Session registers null for skip handlers

```typescript
navigator.mediaSession.setActionHandler("previoustrack", null);
navigator.mediaSession.setActionHandler("nexttrack", null);
```

This explicitly disables the skip buttons in OS media controls (Chrome shows them grayed out). The context already exposes `previousTrack` and `playNext`, so these should be wired through.

## Route-change stops all playback

```typescript
useEffect(() => {
  if (previousPathnameRef.current !== pathname) {
    previousPathnameRef.current = pathname;
    if (audioRef.current) {
      audioRef.current.pause();
      audioRef.current = null;
    }
    clearPlayerState();
  }
}, [pathname]);
```

Every navigation kills the currently playing audio. This contradicts the mini-player UX (persistent bottom bar implies playback continues) and the "continue listening" feature (which expects a session to survive navigation). If stopping on route change was intentional for the old preview-only mode, it now conflicts with the richer player that has queue management and history.

## `endedCounter` effect captures stale state

The effect that handles track-ended logic uses only `[endedCounter]` as its dependency array (with an eslint-disable comment). At the moment a track ends, it reads `repeatMode`, `queue`, `shuffleMode`, `currentTrack`, and calls `startPlayingTrack` and `clearPlayerState`. If any of these changed since the effect was last set up (user toggles repeat mode mid-song), the effect acts on outdated values. With React 19's concurrent rendering, this is more likely to manifest.

## Sleep timer side effects inside updater

```typescript
setSleepTimerRemaining((prev) => {
  if (prev === null || prev <= 1) {
    if (audioRef.current) {
      audioRef.current.pause();  // side effect in updater
      setIsPlaying(false);       // nested setState in updater
    }
    ...
    return null;
  }
  return prev - 1;
});
```

React's state updater functions are expected to be pure. In concurrent mode, React may invoke them more than once. Pausing audio and clearing intervals inside the updater means those side effects could fire multiple times. A safer pattern: decrement the counter in the updater (pure), then in a separate effect check `sleepTimerRemaining <= 0` and perform the side effects there.

## `ContinueListening` external store subscription is a no-op

```typescript
function subscribeToContinueListening() {
  return () => {};
}
```

`useSyncExternalStore` requires the subscribe function to actually notify when the data changes, otherwise the component renders once with whatever `getSnapshot` returns at mount time and never updates. Since `addToPlaybackHistory` modifies localStorage synchronously in the same tab, the ContinueListening component on the homepage won't show a track played in the current session unless the user navigates away and back (triggering a remount).

## Theme transition performance

```css
html.theme-transitioning,
html.theme-transitioning *,
html.theme-transitioning *::before,
html.theme-transitioning *::after {
  transition: background-color 0.3s ease, border-color 0.3s ease, color 0.2s ease,
    box-shadow 0.3s ease !important;
}
```

On the discover page with 40+ browse cards (each with pseudo-elements), this forces the browser to set up and composite hundreds of simultaneous transitions. The `prefers-reduced-motion` override is present, but for users without it on a heavy page, expect a visible frame drop. A scoped approach (transition only on `html` and immediate children, or only on `background-color` on `body`) would avoid the fan-out.

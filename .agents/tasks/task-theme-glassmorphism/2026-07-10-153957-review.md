# Glassmorphism token system and theme-adaptive components

The change replaces hard-coded, theme-unaware glass utility classes with a token-driven glass system that adapts to light and dark mode through CSS custom properties. Six new design tokens (`--surface`, `--glass-bg`, `--glass-border`, `--shadow-color`, `--ring`, `--surface-hover`) back four purpose-specific glass utilities (`.glass`, `.glass-search`, `.glass-dropdown`, `.glass-player`). Components that previously forced a single appearance (dark-only MiniPlayer, hardcoded active filter text) now inherit colors from the theme.

A universal `*` selector applies a 200ms transition on color, background-color, border-color, box-shadow, and outline-color to every element and pseudo-element, giving the entire app smooth theme switching without per-component transition declarations.

**Watch for:** The universal `*` transition rule will animate `background-color` on skeleton loaders that use `animate-pulse`, producing sluggish pulses instead of crisp blinks (confirmed). The `.glass-search` utility carries a hardcoded `border-top-color: rgba(255, 255, 255, 0.3)` that doesn't adapt between themes (confirmed). No `prefers-reduced-motion` opt-out exists for the universal 200ms transition (confirmed).

## High-level view

The token architecture eliminates the old pattern of defining `.glass` once and then fighting it with `html.dark .glass` and `:root .glass` specificity overrides. Each variant now reads from shared custom properties that flip atomically on theme change. The only variant-specific dark override remaining is `.glass-player`, which bumps opacity in dark mode for readability over artwork blurs.

The universal `*` transition is a one-line way to deliver smooth theme switching everywhere, but it trades precision for blast radius. `background-color` in that rule collides with Tailwind's `animate-pulse` keyframes and makes interactive `:active` states feel 200ms late on any element without its own explicit transition declaration.

Component fixes are narrowly targeted: MiniPlayer becomes theme-adaptive, SearchFilters gets a semantically correct contrast pairing, and SearchBar/ShareMenu swap static backgrounds for glass utilities.

<summary>Issues (3)</summary>

1. **Universal `*` transition interferes with `animate-pulse`** — The `transition-property` on `*` includes `background-color`, which fights Tailwind's skeleton loader animation across dozens of loading screens. Scope the transition to semantic containers or add `[class*="animate-"] { transition: none; }` to exclude animated elements.
2. **Hardcoded border-top highlight in `.glass-search`** — `border-top-color: rgba(255, 255, 255, 0.3)` is theme-unaware. In light mode it's less visible than the sides (inverted from intent); in dark mode it creates a highlight edge. Replace with a CSS variable or remove to let `var(--glass-border)` govern all sides.
3. **No `prefers-reduced-motion` opt-out** — Users with reduced-motion preferences still receive 200ms transitions on every color/background/shadow change across all elements. Add `@media (prefers-reduced-motion: reduce) { *, *::before, *::after { transition-duration: 0ms; } }`.

## Universal transition and animation interference

```css
*,
*::before,
*::after {
  transition-property: color, background-color, border-color, box-shadow, outline-color;
  transition-duration: 200ms;
  transition-timing-function: ease;
}
```

`background-color` is the property Tailwind's `animate-pulse` keyframes manipulate. The codebase has dozens of skeleton loading elements (`h-5 w-16 animate-pulse rounded-lg bg-border` in every `loading.tsx`). The 200ms ease transition smooths over what should be a crisp blink between keyframe steps, producing a washed-out pulse.

The same issue affects interactive elements: any button or link that changes background on `:active` will feel 200ms late unless it declares its own `transition` (which overrides the `*` selector by specificity). `.transition-premium` already does this, but plain Tailwind `transition-colors` utilities also carry their own duration and win — so the actual blast radius is elements with *no* explicit transition class that still change colors on interaction.

A `@media (prefers-reduced-motion: reduce)` block should zero out the duration for accessibility. Consider narrowing the selector to `body` and semantic containers, or excluding elements that carry `animate-*` classes.

## Glass token architecture and the border-top highlight

```
:root      → glass-bg: rgba(255,255,255,0.7), glass-border: rgba(255,255,255,0.5)
html.dark  → glass-bg: rgba(24,24,27,0.75),  glass-border: rgba(255,255,255,0.08)
```

The one anomaly is `.glass-search`:

```css
.glass-search {
  border: 1px solid var(--glass-border);
  border-top-color: rgba(255, 255, 255, 0.3); /* hardcoded */
}
```

In dark mode, `--glass-border` is `rgba(255,255,255,0.08)` and the top jumps to `0.3` — a frost highlight, presumably intentional. In light mode, `--glass-border` is `rgba(255,255,255,0.5)` and the top drops to `0.3` — the top is *less* visible than the sides. This inversion suggests the highlight value should be a variable that adapts per theme (brighter white in dark, a subtle dark line in light).

# Editorial homepage, universal search, and cinematic artist experience

This change replaces the static search-focused homepage with a full editorial magazine layout (hero banner, featured artists carousel, trending grid, editorial picks, new releases, mood-based recommendations), redesigns the search experience with categorized results, artwork-enriched autocomplete, filter chips, and an AI discovery panel, and rebuilds the artist page as a cinematic parallax hero with sticky sub-navigation, top tracks, timeline discography, similar artists network, video player, expandable bio, and an insights sidebar. The implementation introduces 22 new components and rewrites 2 page-level files, adding roughly 3,400 lines across the three feature sets.

**Watch for:** The AIDiscoveryPanel fetches directly from the iTunes API on the client, bypassing the existing service layer and rate-limit patterns used elsewhere (confirmed). The ArtistHero parallax re-renders the entire component on every scroll frame via `setState` (confirmed). The ArtistNav scroll-spy attaches a second scroll listener inside a `useEffect` that duplicates the `useSyncExternalStore` pattern defined two lines above it (confirmed). The `PersonalizedSection` silently swallows fetch errors, leaving stale mood data with no user-facing feedback (confirmed).

## High-level view

The homepage's editorial layout is entirely server-rendered: the `EditorialHomepage` async component fetches trending songs, new releases, and mood recommendations in parallel, then delegates to presentational components. The `PersonalizedSection` client component that handles mood switching silently swallows errors and leaves stale data visible without any indication to the user that the mood change failed.

Search adds a multi-source categorized view (`SearchResultsLayout`) driven by a top-result detection heuristic that prioritizes exact artist name matches. The `AIDiscoveryPanel` is the outlier: it makes raw client-side calls to the iTunes API, duplicating search logic that exists in the service layer and bypassing server-side caching. This creates an inconsistency in how API calls are managed and may hit rate limits under moderate usage.

The artist page parallax in `ArtistHero` uses `useState` + a scroll listener, meaning the entire component tree (two `<Image>` elements, `ArtistActions`, stats pills, genre tags) re-renders on every frame. The `ArtistNav` has a redundant scroll-listener architecture: `useSyncExternalStore` for visibility, then a separate `addEventListener` in a `useEffect` for section detection, when both could share a single subscription.

1. **AIDiscoveryPanel bypasses service layer** — Client-side fetch to `itunes.apple.com` directly, skipping the existing `searchTracks` service and its caching. Route through the existing API layer or a dedicated endpoint.
2. **ArtistHero per-frame re-render** — `setScrollY(window.scrollY)` on every scroll event triggers full component re-renders. Use a ref + CSS custom property or `requestAnimationFrame` throttling to avoid re-rendering the image-heavy tree.
3. **Redundant scroll listeners in ArtistNav** — `useSyncExternalStore` provides `scrollY` for visibility, then a separate `addEventListener` in `useEffect` provides scroll position for section detection. Consolidate into one subscription.
4. **Silent error swallowing in PersonalizedSection** — The catch block is empty; the user sees stale tracks from the previous mood with no feedback. Surface an error state or toast.
5. **Missing abort/cleanup in AIDiscoveryPanel** — `useEffect` triggers `fetchDiscovery` on every `query`/`activeMood` change but has no `AbortController` cleanup, so stale responses from a previous mood can overwrite current state if the network is slow.
6. **ArtistVideos lacks error state** — The catch block is empty and the only fallback is returning `null` when `videos.length === 0`. If the YouTube API fails, the section disappears silently.

## AIDiscoveryPanel direct iTunes fetch

The `AIDiscoveryPanel` constructs a search URL directly against `itunes.apple.com/search` from the client:

```typescript
const searchTerm = `${mood} ${query}`;
const response = await fetch(
  `https://itunes.apple.com/search?term=${encodeURIComponent(searchTerm)}&media=music&entity=musicTrack&limit=8`
);
```

Every other component in the codebase fetches through the server layer (`/api/*` routes or server components calling `lib/itunes.ts`). This direct client-side call skips server-side caching, doesn't benefit from the existing error normalization in the service layer, and exposes the raw iTunes API surface to CORS behavior differences across browsers. The `PersonalizedSection` component shows the correct pattern: it calls `/api/recommendations?mood=...` which routes through the server.

The component also lacks an `AbortController` in its `useEffect`. When `query` or `activeMood` changes rapidly, multiple in-flight requests race without cancellation. The last response to arrive wins regardless of whether it corresponds to the current UI state.

## ArtistHero scroll-driven re-renders

```typescript
useEffect(() => {
  const handleScroll = () => {
    setScrollY(window.scrollY);
  };
  window.addEventListener("scroll", handleScroll, { passive: true });
  return () => window.removeEventListener("scroll", handleScroll);
}, []);

const parallaxOffset = scrollY * 0.4;
```

Every scroll event calls `setScrollY`, which re-renders the component. The parallax transform is purely visual and applied via inline `style={{ transform: ... }}` on a single div. This is a textbook case for using a ref to mutate the DOM directly or writing the scroll offset to a CSS custom property, avoiding React's reconciliation entirely.

## ArtistNav dual-subscription architecture

`ArtistNav` calls `useSyncExternalStore` to get `scrollY` (used only for `const isVisible = scrollY > 500`). Then, inside a `useEffect`, it attaches a *second* scroll listener that reads `window.scrollY + 200` to determine the active section. Both subscriptions fire on every scroll event. The section-detection logic could read the same `scrollY` value from the store, or the visibility check could be folded into the `useEffect` handler, eliminating one subscription.

## PersonalizedSection error path

```typescript
} catch {
  // Keep existing tracks on error
}
```

When a mood change fails, the user sees tracks from the *previous* mood with no indication that the new mood failed to load. The loading state resets to `false`, so there's no spinner either. This creates a confusing UX where tapping "energetic" leaves "chill" tracks on screen. A minimal fix: set an error flag and show a brief inline message, or revert `activeMood` to the previous value on failure.

## Discography timeline scroll discoverability

The timeline view sets `style={{ minWidth: "max-content" }}` with `overflow-x-auto` on the parent. For artists with many albums, this creates a wide scrollable area with no visible scroll indicator or scroll-snap behavior. On mobile, the horizontal scroll is discoverable only by accident. A scroll-snap or visible left/right arrows would improve discoverability (likely — not verified on a device, but the pattern is common enough to flag).

## ArtistNav aria-current semantics

```typescript
aria-current={activeSection === item.id ? "true" : undefined}
```

`aria-current="location"` would be semantically more precise for in-page navigation anchors than the generic `"true"` value. Minor (confirmed — the attribute works, it's just less specific than it could be).

# Browser-local favorites via localStorage and React context

This change adds a favorites feature backed by browser localStorage. Users toggle a heart button on track cards to persist songs, view them on a dedicated `/favorites` page, and navigate there via a header link on every page. The implementation introduces a localStorage helper module, a React context powered by `useSyncExternalStore` for hydration-safe state, and a `FavoriteButton` that intercepts clicks within the existing `Link`-based card.

**Watch for:** JSON.stringify comparison on every render in `getSnapshot` creates O(n) serialization cost that scales with favorites count (confirmed). Cross-tab sync is absent ŌĆö the `storage` event is not subscribed to, so two tabs will diverge (confirmed). TrackCard's conversion to a client component opts the entire card out of server rendering (confirmed ŌĆö likely acceptable for this app's scale, but worth noting as a trade-off).

## High-level view

The persistence layer in `lib/favorites.ts` is thin: four functions wrapping `localStorage.getItem`/`setItem` with a try-catch guard on reads but not writes. It stores the full `ItunesTrack` object array, which means localStorage payload grows linearly with the number of favorites and includes fields the favorites feature never uses (preview URLs, genre, release date).

The context layer uses `useSyncExternalStore` to avoid the setState-in-useEffect lint violation. The `getSnapshot` function performs a JSON round-trip comparison on every call to detect changes, which is the primary performance concern in this change. The notify/subscribe pattern is internal-only and does not listen to the browser `storage` event, so multi-tab consistency is impossible.

TrackCard became a client component to access the favorites context. The FavoriteButton uses `stopPropagation` and `preventDefault` to coexist with the parent `Link` without triggering navigation.

Navigation to `/favorites` is added in three places (home, detail, favorites page itself). The home page uses an `<a>` tag while the others use Next.js `<Link>`, creating an inconsistency ŌĆö `<a>` triggers a full page reload and tears down client-side state like the audio player.

<details>
<summary>Issues (5)</summary>

1. **getSnapshot JSON.stringify on every render** ŌĆö `getSnapshot` serializes the full favorites array to compare with the previous snapshot on every React render cycle. Replace with a version counter incremented on mutations, or move the equality check to only run after `notify()` is called.
2. **No cross-tab sync** ŌĆö The context does not listen to the `window` `storage` event, so changes in one tab are invisible in another until a page reload. Subscribe to `storage` events in the `subscribe` callback and call `notify()` when the key matches.
3. **Full track objects stored in localStorage** ŌĆö Each favorite stores the entire `ItunesTrack` (14 fields). For 50 favorites this approaches 50KB. Consider storing only `trackId` and hydrating on demand, or at minimum trimming to the fields needed for display (id, name, artist, artwork, duration, collection).
4. **Inconsistent nav link element** ŌĆö `app/page.tsx` uses `<a href="/favorites">` while `app/track/[id]/page.tsx` and `app/favorites/page.tsx` use `<Link href="/favorites">`. The `<a>` causes a full page reload, losing client-side state (audio player, etc.). Import `Link` from `next/link` in `app/page.tsx` and use it.
5. **No localStorage quota handling** ŌĆö `addFavorite` does not catch `setItem` failures (e.g., quota exceeded). A try-catch around the write with silent degradation is needed to avoid an unhandled exception.

## getSnapshot serialization cost

The `getSnapshot` function passed to `useSyncExternalStore` is called on every render of any component that consumes the context ŌĆö which includes every visible `TrackCard` via `isFavorite`:

```typescript
const getSnapshot = useCallback(() => {
  const current = getFavorites();
  if (JSON.stringify(current) !== JSON.stringify(snapshotRef.current)) {
    snapshotRef.current = current;
  }
  return snapshotRef.current;
}, []);
```

`getFavorites()` parses localStorage JSON, then `JSON.stringify` runs on both the parsed result and the previous snapshot. For N favorites with 14 fields each, this is two full serializations per render per subscriber. At 20 favorites across a grid of 12 visible cards, that's 24 serializations per interaction. The fix is to use a monotonically increasing version counter bumped in `notify()` and let the snapshot update only there ŌĆö `getSnapshot` returns `snapshotRef.current` without re-reading localStorage.

## Cross-tab divergence

The implementation only notifies listeners when `addFavorite` or `removeFavorite` is called within the same tab. The browser's `storage` event (fired when localStorage changes in _other_ tabs) is never subscribed to. Adding a `window.addEventListener("storage", ...)` inside the `subscribe` callback and calling `notify()` when the event key matches `"music-search-favorites"` would close this gap.

## localStorage payload growth

Each `ItunesTrack` object contains fields like `previewUrl`, `trackViewUrl`, `collectionViewUrl`, `releaseDate`, `kind`, and `wrapperType` that the favorites page never displays. A favorites list of 100 tracks could reach 100-200KB depending on string lengths. The favorites page only needs artwork, name, artist, collection, and duration ŌĆö five fields instead of fourteen.

## Navigation element inconsistency

In `app/page.tsx`, the favorites link is a plain `<a>`:

```tsx
<a href="/favorites" className="..." aria-label="Favorites">
```

This triggers a full navigation, tearing down the `AudioPlayerContext` and stopping any in-progress playback. The file doesn't import `Link` from `next/link`, but server components can render `<Link>` ŌĆö adding the import and switching to `<Link>` matches what the other pages do.

## No write-failure handling in localStorage helpers

`addFavorite` and `removeFavorite` call `localStorage.setItem` without a try-catch. If the user's storage is full (private browsing in some browsers allocates minimal quota), the exception propagates up through the React event handler unhandled. `getFavorites` defensively catches parse errors on reads, but the write path has no equivalent guard.

# Browser-local favorites with useSyncExternalStore (v2)

This revision addresses four of the five issues from the v1 review. The `getSnapshot` function no longer serializes on every render — it returns `snapshotRef.current` directly. Cross-tab sync is wired via a `storage` event listener in the `subscribe` callback. Navigation in `app/page.tsx` is now a proper `<Link>`. localStorage writes are wrapped in try-catch. The one explicitly accepted trade-off is storing full `ItunesTrack` objects rather than trimming to display-relevant fields.

**Watch for:** Silent failure on write quota — the user receives no feedback when `addFavorite` fails (confirmed, minor). `isFavorite` performs a linear scan via `favorites.some()` on every TrackCard render, which is O(favorites × visible cards) — switching to a `Set<number>` derived from the snapshot would reduce per-card cost to O(1) (confirmed, low severity at expected scale).

**Verdict**: APPROVED

## High-level view

The persistence layer defensively catches both read and write failures. Storing full track objects avoids a re-fetch requirement on the favorites page and keeps the code simple at the cost of larger localStorage payloads.

The context layer's `getSnapshot` returns the ref directly. Mutations update the ref and call `notify()`, and the `storage` event handler keeps other tabs in sync.

Each TrackCard calls `isFavorite(track.trackId)`, which runs `favorites.some(...)` — O(N) per card, O(N×M) per grid render. A `Set<number>` memoized from the snapshot would eliminate this, but it's unlikely to cause visible jank below ~100 favorites.

The header nav heart SVG is duplicated across three pages; a shared component would reduce maintenance surface.

## Silent write failure UX

`addFavorite` in `lib/favorites.ts` catches the `setItem` exception and returns `void`:

```typescript
try {
  localStorage.setItem(STORAGE_KEY, JSON.stringify(favorites));
} catch {
  // Silently handle quota exceeded or other write failures
}
```

After this catch, `notify()` is still called in the context layer, which re-reads localStorage and updates `snapshotRef`. Because the write failed, the re-read returns the old value, so the UI snaps back — but only after the button briefly shows the toggled state. This creates a flicker rather than a clear error. Returning a success/failure signal from the helpers (or throwing a domain-specific error the context can catch) would let the UI suppress the toggle or surface a toast.

## isFavorite cost at scale

The context exposes `isFavorite` as:

```typescript
const isFavorite = useCallback(
  (trackId: number) => {
    return favorites.some((fav) => fav.trackId === trackId);
  },
  [favorites]
);
```

Every `TrackCard` calls this on render. A search result page with 20 cards and a favorites list of N tracks means 20×N comparisons per render pass. Replacing the linear scan with a `useMemo`-derived `Set<number>` keyed on the `favorites` reference would make each check O(1):

```typescript
const favoriteIds = useMemo(() => new Set(favorites.map(f => f.trackId)), [favorites]);
const isFavorite = useCallback((id: number) => favoriteIds.has(id), [favoriteIds]);
```

At realistic scale (< 100 favorites) this won't cause noticeable jank — a polish item, not a blocking concern.
